# 竞品销量纯视觉自动化统计系统

## 产品需求文档与技术落地规范 (V5.0 生产级里程碑版)

> **产品定位：** 本系统是一款专为电商运营打造的**"即插即用型"竞品微观数据监视器**。系统采用**纯视觉解耦架构（Visual RPA）**，完全切断与目标 App 底层代码及无障碍节点（Accessibility）的交互。通过引入白天随动触发机制、本地人机协同解围与多模态大模型判定，实现对竞品大店"24小时内实时销量流水"的物理级、高隐蔽性精准捕获。

## 1. 产品定义与核心商业价值

### 1.1 传统反爬痛点与视觉 RPA 解决方案

在电商大数据对抗中，传统的逆向接口（API 破解）与无障碍服务（Accessibility XML 遍历）面临高额的维护成本与极高的封号风险。本系统全面改用**纯视觉解耦方案**：

| 维度 | 传统自动化方案 (逆向/无障碍) | 纯视觉自动化方案 (本系统) |
|------|---------------------------|--------------------------|
| **底层对抗** | 极易被反爬 SDK 识别到 Hook、调试特征或虚拟显示器 | 将手机屏幕视为纯画布，不读取任何底层节点，风控识别难度呈几何级上升 |
| **界面改版** | 节点 ID 或加密算法微调即导致全盘崩溃 | 具备"界面布局改版防护网"，容错率极高 |
| **数据颗粒度** | 只能抓取滚动 30 天的宏观销量，无法规避"大单滑出统计周期"导致的负数陷阱 | **直击微观流水**，精准捕获 24 小时内用户的下单事件，彻底还原真实转化 |

### 1.2 产品核心价值

1. **绝对安全的资产保护：** 结合 KernelSU 内核级隐匿与真人随动划动，将采集账号被封禁（关黑屋）的概率降至无限接近于零。
2. **极低的使用门槛：** 随动触发模式，用户"插上手机 → 启动脚本 → 顺畅跑完 → 拔线走人"，无需机房群控硬件开销。
3. **零资费本地化：** 核心推理算力完全部署于本地 Apple Silicon 架构上，无任何云端大模型 API 消耗资费。

## 3. 全局架构与并发机制设计

为解决高帧率视频流与多模态大模型（百毫秒级推理）之间的物理速率冲突，中枢控制端必须采用严格的**生产者-消费者（Producer-Consumer）双线程解耦架构**。

```text
┌──────────────────────────────────────────────────────────────────┐
│                        [ Android 手机 ]                          │
│                              │                                   │
│                              ▼ (无特征投屏协议视频流)             │
└──────────────────────────────────────────────────────────────────┘
                               │
┌──────────────────────────────────────────────────────────────────┐
│           电脑控制端 (环境强断言 + KernelSU 内核隐匿校验)           │
│                              │                                   │
│  【线程 A：生产者 (RPA 控制与投屏接收)】                           │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ - 实时接收 1080P 帧矩阵                                  │   │
│   │ - 视觉干扰剔除（直播悬浮窗/促销GIF 动态涂白）             │   │
│   │ - 注入带触控压力/形变量的贝塞尔仿生手势流                 │   │
│   │ - 顺次压入有界阻塞队列                                   │   │
│   └──────────────────────┬──────────────────────────────────┘   │
│                          │                                      │
│                          ▼                                      │
│             [ 图像待处理阻塞队列 (容量固定上限: 10 帧) ]          │
│                          │                                      │
│                          ▼                                      │
│  【线程 B：消费者 (AI 视觉解析)】                                 │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │ - 提取大图 -> OpenCV 网格切分 -> 信息熵白块过滤           │   │
│   │ - 本地 Qwen2.5-VL-7B 多模态结构化提取                     │   │
│   │ - 时间归一化清洗 (正则匹配换算绝对日期)                   │   │
│   │ - 24小时边界检测 -> 触发熔断 -> 关闭弹窗 -> 流转下一单品  │   │
│   └──────────────────────┬──────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
                               │
                               ▼ (触发异常/验证码 -> 本地蜂鸣挂起 -> 人工解锁回车复活)
                               │
                               ▼ (标准 JSON 结构体)
┌──────────────────────────────────────────────────────────────────┐
│    [ MySQL 数据库 (按月物理分区) ] ◄── [ 前端可视化趋势大屏 ]     │
│    competitor_realtime_orders          (Vue3 + ECharts)          │
│    rolling_sales_history                                         │
└──────────────────────────────────────────────────────────────────┘
```

### 3.1 线程 A：生产者 (RPA 控制与投屏接收)

- **职责：** 通过高速无特征投屏协议，实时接收物理手机的 1080P 帧矩阵。
- **视觉干扰剔除：** 实时对截幅矩阵执行动态变动率检测。若发现屏幕右下角或固定区域存在高频像素变动（判定为直播悬浮窗或促销 GIF），自动对该局部坐标矩阵执行"物理涂白"（遮罩遮挡），防止其破坏下游 OpenCV 的卡片切分算法。
- **手势注入：** 负责依据贝塞尔曲线向手机注入带有触控压力、形变量的仿生手势流。

### 3.2 线程 B：消费者 (AI 视觉解析)

- **职责：** 从有界队列中提取干净的帧图像，执行网格切分、去重与大模型结构化清洗。
- **核心约束：** **图像待处理队列必须为有界阻塞队列（Bounded Blocking Queue），最大容量上限固定为 10 帧**。当队列满时，生产者线程强制进入 `WAITING` 状态，停止一切手势注入与物理截图，彻底根除因大模型推理卡顿导致的内存飙升与 OOM（内存溢出）危机。

## 4. 核心功能模块详细设计

### 4.1 环境强断言预检模块 (Pre-flight Gate)

系统在正式运行业务首帧前，必须在微秒级内对底层执行四项硬核断言。任何一项不通过，立即在终端红字报错并完全终止程序：

| 序号 | 断言项 | 检查内容 | 失败动作 |
|------|--------|---------|---------|
| 1 | **连接唯一性断言** | 校验 `adb devices` 返回的设备列表，必须能且仅能看到一台合法的 Android 设备 | 红字报错退出 |
| 2 | **屏幕状态自愈** | 读取设备亮屏状态，若处于熄灭状态，强制注入系统唤醒键，并向特定物理坐标发送滑动解锁流 | 自动恢复后重检 |
| 3 | **App 版本匹配断言** | 读取目标电商 App 的 `VersionCode`，必须与配置文件中预设的 OpenCV 绝对坐标切分模板完全一致，防范静默改版 | 红字报错退出 |
| 4 | **底层隐匿校验** | 检测设备 KernelSU 或 APatch 的内核防护状态，确保反爬 SDK 无法扫描到任何调试特征 | 红字报错退出 |

### 4.2 动态入口寻址模块 (Dynamic Entry Locating)

脚本控制手机进店并点击进入某一具体商品详情页后，不依赖固定的死坐标，而是利用轻量化 PaddleOCR 在屏幕中下部实时检索"流量探针标签"。

**特征匹配词库（支持正则）：**

```python
ENTRY_FEATURES = [
    r'超\d+人加购',
    r'\d+人感兴趣',
    r'近\d+天\d+\+人逛过',
    r'近期销量飙升',
]
```

**定位动作：**
只要命中上述任一特征，提取其文本矩形的中心坐标 `(x, y)`，叠加符合二维高斯分布的随机偏移量后执行点击，唤醒"实时销量"列表弹窗。若未命中，系统自动降级抓取页面顶部常规的"月销"大盘数据，然后退出当前商品。

### 4.3 懒加载白块视觉过滤机制 (Lazy Loading Filter)

在弹窗列表滚动或快速翻页时，由于网络延迟，手机画面常会出现"纯白块"或"正在加载"的占位符。

**首道视觉拦截网：**
OpenCV 在执行卡片矩形切分后，在送入大模型前，立即对"主图区域"和"销量区域"执行特征检验。

**指标判定：**

| 指标 | 正常商品帧 | 白块/未加载帧 |
|------|-----------|--------------|
| 图像信息熵（Entropy） | > 4.0 | < 1.0 |
| 灰度方差（Variance） | > 500 | < 10 |

**熔断处理：**
命中上述指标（纯色占位符特征），该帧卡片数据**在当前生命周期内立即物理销毁**。拒绝将其送入下游，不计入去重队列，不调用本地大模型，保护算力资源不被空转浪费。

#### 4.3.1 骨架屏升级防御：SSIM + Sobel 梯度双重拦截（V5.0 优化）

**潜在工程瓶颈：**
目前的电商 App 升级了加载动效，未加载完成的商品卡片不再是纯白块，而是带有灰色呼吸渐变**骨架屏（Skeleton Screen）**，或带有旋转的官方 Logo 加载动画。此时画面的信息熵会超过 2.5，灰度方差也会增大，原有的纯白块拦截网直接失效。

**优化方案：SSIM 模板匹配 + 索贝尔边缘算子**

- **骨架屏特征库（Template Library）：** 提前截取 2-3 张目标 App 在低网速下的标准加载骨架屏卡片、旋转动画的局部 ROI，在本地内存中建立 `Invalid_Templates` 矩阵。
- **结构相似性（SSIM）快速前置判定：** OpenCV 切分出商品卡片后，与模板库计算 SSIM。若 `SSIM > 0.85`，直接判定为未加载完的无效卡片，跳过后续所有流程。
- **索贝尔梯度检查（Edge Gradient Check）：** 对销量区域（区域 C）计算 Sobel 梯度方差：

```python
grad_x = cv2.Sobel(gray_roi, cv2.CV_16S, 1, 0)
variance_grad = np.var(grad_x)
```

若 `variance_grad` 低于设定阈值（说明区域内没有清晰的文字边缘，全是模糊的渐变色块），直接触发熔断销毁该帧，确保送入大模型的每一张图都是 100% 渲染完毕的超高质量真数据。

### 4.4 视觉多模态数据提取 (Multimodal Extraction)

实时销量弹窗内由于排版松散，传统 OCR 易发生错行、错位。系统全权调用本地常驻运行的 **Qwen2.5-VL-7B-Instruct** 大模型执行高阶语义提取。

**指令集约束 (System Prompt)：**

```text
System: 你是一个电商数据结构化清洗专家。
请提取图片中"实时销量"列表的数据，忽略头像。
将每一行提取为一个对象，严格输出如下标准的 JSON 格式，不要包含任何解释或 Markdown 标记：
{
  "summary": {
    "cart_adds": "顶部汇总用户加购人数文本",
    "growth": "周销量上涨倍数文本"
  },
  "orders": [
    {
      "buyer": "买家脱敏ID",
      "is_repeat": true或false（识别是否带有"回头客"标签）,
      "sku": "购买的商品款式或课程名称",
      "time_str": "下单时间文本"
    }
  ]
}
```

**大模型部署规范：**

- 电脑端后台常驻 Ollama 进程，加载 Qwen2.5-VL-7B-Instruct（4-bit 量化）。
- `temperature: 0` 保证结果确定性，`num_predict: 512` 限制生成长度。
- 启动时执行预热握手 `keepalive: -1`，结束后释放 `keepalive: "0s"`。

#### 4.4.1 视觉状态机与增量推理优化（V5.0 优化）

**潜在工程瓶颈：**
当"实时销量"弹窗向下滑动时，每一屏画面可能包含 30%~50% 与上一帧重叠的旧订单。如果直接把整屏新截图丢给 VLM 推理，会造成：TTFT（首字延迟）重复叠加，Apple Silicon 长时间高负载引发 Thermal Throttling（发热降频），VLM 对重叠区域的文字重复解码浪费本地算力。

**优化方案：局部差分感知 + KV-Cache 状态保持**

- **局部差分感知（Image Diff）：** 消费者线程从阻塞队列中提取弹窗帧时，不再将全图送入 VLM。使用 OpenCV 对弹窗的可滚动剪切区（Scrollable ROI）计算像素差分绝对值（SAD）。顺次计算滑动的像素位移 $\Delta y$，直接把画面底部 $\Delta y$ 像素高度的新涌入增量区域裁剪出来，仅将增量切片送入 Qwen-VL 推理。
- **KV-Cache 状态保持（上下文复用）：** 调用本地 VLM API 时，开启系统级的 `slot_id` 或上下文缓存复用。保持 System Prompt 和主图结构部分的 KV-Cache 不被清空，单次推理耗时可从 400ms 骤降至 80ms 左右。

#### 4.4.2 显存物理红线与内存锁定（V5.0 优化）

**隐性盲区：**
RTX 4060 的 8GB 显存极易受到外部环境的"显存强行驱逐（VRAM Eviction）"。如果运行期间用户电脑打开了带有硬件加速的浏览器（如 Chrome/Edge 播放视频）或背景运行了 3D 渲染，Ollama 的显存会被系统动态切走，直接导致推理产生不可预知的延迟甚至闪退。

**技术补丁：**

- **并发锁死：** 设置环境变量 `OLLAMA_NUM_PARALLEL=1`，彻底禁止 Ollama 同时处理多个请求，将显存独占绑定给当前推理任务。
- **内存锁定（Memory Lock）：** 在 Ollama 启动参数中追加 `OLLAMA_USE_MLOCK=1` 或 CUDA 级 `CUDA_VISIBLE_DEVICES` 隔离，强制模型权重驻留显存，阻止操作系统在内存压力下将 KV-Cache 或模型参数 Swap 到共享内存。
- **动态边界裁剪前置：** 图像在二分之一降采样前，必须先执行"动态边界裁剪"——只保留商品卡片区域（ROI），切掉背景大白边。双重降低序列长度（Sequence Length），减少显存峰值占用。

#### 4.4.3 异构算力深度分流：PaddleOCR 多进程隔离（V5.0 优化）

**核心盲区（致命）：**
Python 的全局解释器锁（GIL）限制。如果 PaddleOCR 仅仅是在一个单独的线程（Thread）中调用，由于它是一个重度消耗 CPU 的计算密集型任务，它在运行时会长时间霸占 GIL 锁。这会导致生产者线程（接收投屏视频流）以及 RPA 触控注入线程发生严重的系统级卡顿和掉帧，有界队列会因锁死而无法消费。

**技术补丁：多进程（Multiprocessing）替代多线程**

- **进程级物理隔离：** 必须将 PaddleOCR 部署在一个独立的**子进程**中，通过 `multiprocessing.Queue` 与主进程通信。这样才能真正绕过 GIL 锁，让 i5-12600KF 的多核心实现物理级并行——性能核跑 OCR 子进程，能效核跑主 RPA 调度。
- **内存显式回收：** `cv2.destroyAllWindows()` 仅仅是关闭显式渲染窗口。释放内存必须对 NumPy 矩阵显式执行 `del frame` 或使用预分配内存池复用，防止内存碎片累积。
- **线程 vs 进程职责表：**

| 组件 | 运行方式 | 原因 |
|------|---------|------|
| 生产者（帧接收 + 手势注入） | 独立线程 | I/O 密集型，GIL 影响小 |
| 消费者（OpenCV 切分 + Qwen-VL 推理） | 独立线程 | 推理阶段释放 GIL（Ollama HTTP 调用） |
| **PaddleOCR（文本定位）** | **独立子进程** | **CPU 密集型，必须物理隔离 GIL** |

#### 4.4.4 Qwen-VL 视觉切片（Tile）格栅数量裁剪（V5.0 优化）

**性能盲区：**
Qwen2.5-VL 引入了动态分辨率机制。如果直接将一整屏 1080P 的复杂弹窗截图喂给它，模型内部的 Vision Transformer（ViT）会自动将其切分为大量的局部小方块（Tiles）进行并行计算。这会导致图像的 Token 数量指数级暴增，直接撑爆 RTX 4060 的 8GB 显存红线。

**优化规范：限制推理上下文网格尺寸**

- **强制下采样：** 调用本地 Ollama 接口时，将弹窗区域等比例下采样至最大 **512 像素**边长。这既不影响识别"X小时前下单"这类清晰文本，又能将每次推理的显存开销稳定控制在固定极低水位。
- **max_tiles 限制：** 在 Ollama 模型配置中限制 `max_tiles` 参数，防止 ViT 自动切分过多局部方块。
- **前置裁剪：** 结合 4.4.1 的增量差分裁剪——只送底部 Δy 增量切片，双重降低 Token 数量：

```python
# 下采样到最大 512px 后再送 VLM
import cv2
h, w = roi.shape[:2]
scale = 512 / max(h, w)
if scale < 1.0:
    roi = cv2.resize(roi, (int(w * scale), int(h * scale)))
```

### 4.5 24小时智能熔断截断机制 (Early Stop Mechanism)

这是确保系统兼顾"极高采集效率"与"类人浅层浏览特征"的核心模块。

1. **时间归一化清洗：** 本地 Python 消费者线程接收到 Qwen-VL 返回的 JSON 数组后，立即通过正则匹配将 `15小时前下单`、`3分钟前` 换算为绝对日期 `YYYY-MM-DD`。

```python
import re
from datetime import datetime, timedelta

def normalize_time(time_str: str) -> str:
    """将相对时间文本归一化为绝对日期 YYYY-MM-DD。"""
    now = datetime.now()
    if match := re.match(r'(\d+)小时前', time_str):
        dt = now - timedelta(hours=int(match.group(1)))
    elif match := re.match(r'(\d+)分钟前', time_str):
        dt = now - timedelta(minutes=int(match.group(1)))
    elif match := re.match(r'(\d+)天前', time_str):
        dt = now - timedelta(days=int(match.group(1)))
    elif '天前' in time_str:
        dt = now - timedelta(days=1)
    else:
        dt = now
    return dt.strftime('%Y-%m-%d')
```

2. **熔断判定：** 在遍历 `orders` 列表时，一旦检测到某条记录的 `time_str` 包含 **"1天前"**、**"2天前"** 或 **"天前"** 关键词，说明 24 小时内的微观实时单流水已全部拿全。

3. **阻断协同：** 消费者线程立即向生产者线程抛出 `STOP_SCROLL_AND_CLOSE` 强阻断信号。生产者接收信号，瞬间终止后续下滑动作，直接点击弹窗右上角的 `X` 按钮关闭弹窗，退出该单品。

> **效率收益：** 该机制使单品耗时从 20 秒压缩至 3 秒，单次跑盘效率提升 5 倍以上，且彻底隐匿于平台"拉取全量列表"的风控审计红线之下。

#### 4.5.1 零点跨天时间清洗状态机（V5.0 优化）

**潜在工程瓶颈：**
如果运营人员在深夜或凌晨临界点（例如 23:30 至 01:00 之间）插入手机启动脚本，相对时间转化会发生严重的日期漂移错误。例如当前系统时间是 2026-05-28 00:30，大模型解析到一条订单是"2小时前下单"。如果单纯用当天日期减去小时，该订单应归属于 2026-05-27。若状态机没有跨天闭环，极易误判为当天订单，导致 24 小时熔断截断机制出现"提早熔断"或"漏抓数据"。

**优化方案：全时段时序状态机（Temporal Epoch State Machine）**

- **时间锚点锁定：** 会话初始化模块启动时，在内存中锁定当前任务启动的绝对时间锚点（基准 Epoch 毫秒数 `task_start_epoch_ms`）。
- **级联时间解析矩阵：**

```python
def normalize_time(time_str: str, task_start_epoch: float) -> tuple[str, bool]:
    """返回 (YYYY-MM-DD, should_stop)"""
    now_ms = task_start_epoch

    if match := re.match(r'(\d+)小时前', time_str):
        dt = datetime.fromtimestamp(now_ms / 1000) - timedelta(hours=int(match.group(1)))
        return dt.strftime('%Y-%m-%d'), False

    if match := re.match(r'(\d+)天前', time_str):
        days = int(match.group(1))
        dt = datetime.fromtimestamp(now_ms / 1000) - timedelta(days=days)
        return dt.strftime('%Y-%m-%d'), True  # 触发 24h 熔断阻断

    if '分钟前' in time_str or '刚刚' in time_str:
        return datetime.fromtimestamp(now_ms / 1000).strftime('%Y-%m-%d'), False

    return datetime.fromtimestamp(now_ms / 1000).strftime('%Y-%m-%d'), False
```

- **断点批次隔离：** 在微观订单流水表中引入 `capture_batch_id`（由启动时间戳生成的 UUID）。确保即使发生极端的跨天运行，也能通过批次 ID 在后端轻易完成数据对齐和重叠度二次校验。

#### 5.4.1 高并发微批次写入优化（V5.0 优化）

**潜在工程瓶颈：**
随动系统在白天高速运转时，多线程消费者会密集触发短平快的单条 INSERT 请求。在 MySQL 按月分区（Range Partitioning）的架构下，高频单条插入会引发分区锁争用（Partition Lock Contention）及 B-Tree 索引页频繁分裂，压满本地数据库的磁盘 I/O 吞吐。

**优化方案：内存双缓冲区（Double Buffering）与批量写入**

- **异步内存桥接：** 消费者线程解析出订单 JSON 并完成时间归一化后，禁止直接请求数据库。系统开辟两个线程安全的内存并发队列 `Buffer_A` 和 `Buffer_B`（交替读写）。
- **微批次刷盘驱动（Micro-Batch Engine）：** 数据先写入 `Buffer_A`。当满足以下任一条件时，触发批量写入：
  - 队列积压元素达到 **50 条**。
  - 距离上一次刷盘时间过去 **5 秒**。
- **批量 Upsert 语法：** 调度器切换读写缓冲区，使用一条 SQL 批量提交，大幅降低事务提交频次与分区表行级锁的持有时间：

```sql
INSERT IGNORE INTO competitor_realtime_orders
  (virtual_id, buyer_mask, sku_name, is_repeat, order_date, capture_batch_id)
VALUES
  (?, ?, ?, ?, ?, ?),
  (?, ?, ?, ?, ?, ?),
  ...;
```

#### 5.4.2 内存缓冲区防蒸发：三维级联刷盘闸门（V5.0 优化）

**边缘盲区：程序非正常死锁或强行拔线时的"数据蒸发"。**
如果随动任务运行到第 29 条数据时，由于手机断连或用户误操作直接关闭了控制台，这 29 条驻留在内存缓冲区的数据就会永久丢失。

**技术补丁：完善触发刷盘的三维级联条件**

| 级联条件 | 触发时机 | 防护场景 |
|---------|---------|---------|
| **条件 1：容量阈值** | 缓冲区积压达到 **30 条** | 正常高速采集 |
| **条件 2：定时器兜底** | 距离上一次刷盘时间超过 **5 秒** | 单店商品过少，数据长期滞留内存 |
| **条件 3：析构逃生** | 捕获系统异常、USB 拔线信号、`Ctrl+C` 退出信号 | 非正常终止时的最后抢救 |

```python
class MicroBatchBuffer:
    def __init__(self, batch_size=30, flush_interval=5.0):
        self._buffer_a = []
        self._buffer_b = []
        self._active = self._buffer_a
        self._lock = threading.Lock()
        self._last_flush = time.monotonic()

    def insert(self, order: dict) -> None:
        with self._lock:
            self._active.append(order)
            if len(self._active) >= 30:
                self._flush()

    def _flush(self) -> None:
        to_flush = self._active
        self._active = self._buffer_b if self._active is self._buffer_a else self._buffer_a
        self._active.clear()
        self._last_flush = time.monotonic()
        self._batch_insert(to_flush)

    def emergency_flush(self) -> None:
        """析构函数 / 信号处理器调用：抢救所有残留数据"""
        with self._lock:
            if self._active:
                self._batch_insert(self._active)
                self._active.clear()
```

## 6. 极致防封与生物识别仿真架构

为了保证白天流量在电商大厂（如阿里聚安全）的生物识别引擎面前无懈可击，系统在底层触控与网络层进行了深度仿真。

### 6.1 硬件级触控属性仿真 (MotionEvent 审计对抗)

**核心痛点：**
原生 ADB 指令下发的触摸事件，其压力值（Pressure）恒定为 1.0，触控面积恒定为 0，这在黑产风控模型眼中是典型的 Bot 特征。

**优化方案：**
放弃原生 `input swipe` 指令。重构驱动，直接向手机内核 `/dev/input/eventX` 写入二进制原始事件序列。完整模拟人类手指肉垫按压屏幕时的非线性对数形变：

```text
DOWN (压力 0 → 0.3) → MOVE (压力动态波动 0.5 → 0.8，面积动态正态形变) → UP (释放前压力衰减)
```

触控生命周期中触控面积（Size/Major/Minor）同步执行随机正态分布微调。

#### 6.1.1 多机型 DPI 动态映射（V5.0 优化）

**潜在工程瓶颈：**
Linux 触控板的绝对坐标系（Digitizer Coordinates）与 Android 屏幕显示的像素坐标系（Viewport Pixels）通常不是 1:1 映射的。例如一部 1080P 的手机，其底层触控芯片的绝对坐标采样率可能是 `0 ~ 32767`。直接写入像素坐标会导致点击错位或滑屏滑不出。

**优化方案：跨层坐标转换矩阵（Affine Transformation）**

- **动态 DPI 预检：** 脚本在 4.1 节环境预检阶段，必须通过 ADB 执行系统级查询，动态获取当前手机的触控事件设备号及最大坐标边界：

```bash
adb shell getevent -p /dev/input/event1  # 获取 ABS_MT_POSITION_X 的 max 值
```

- **仿生事件序列封装：** 在 Python 侧构建坐标转换矩阵。输入贝塞尔曲线的 Viewport 像素坐标 $(X_{pixel}, Y_{pixel})$，自动按比例转换为底层绝对坐标 $(X_{abs}, Y_{abs})$：

```python
def viewport_to_abs(x_pixel: int, y_pixel: int,
                    screen_w: int, screen_h: int,
                    abs_max_x: int, abs_max_y: int) -> tuple[int, int]:
    x_abs = int(x_pixel / screen_w * abs_max_x)
    y_abs = int(y_pixel / screen_h * abs_max_y)
    return x_abs, y_abs
```

- **完整 Linux Evdev 事件流模拟：** 确保写入事件时包含标准的同步信号。一次完整的仿生滑动必须严格输出以下二进制序列：

```text
SYN_REPORT -> EV_ABS (ABS_MT_TRACKING_ID, 随机ID)
  -> EV_ABS (ABS_MT_POSITION_X) -> EV_ABS (ABS_MT_POSITION_Y)
  -> EV_ABS (ABS_MT_PRESSURE, 动态对数变化) -> SYN_REPORT
```

#### 6.1.2 触控仿真路径降级：ADB 持久化管道（V5.0 优化）

**工程盲区：ADB 进程频繁创建的系统开销（Fork Overhead）。**
如果 Python 脚本高频、连续地调用 `os.system("adb shell input...")`，Windows 系统在微秒级内频繁创建和销毁 `adb.exe` 进程会瞬间榨干 CPU 资源，并造成触控轨迹出现严重的突刺和断点。

**技术补丁：持久化 ADB Shell 管道（Persistent Pipe Session）**

- **严禁频繁开关 ADB 进程。** 必须在初始化阶段与手机建立一个持久化的 ADB 交互式 Shell 管道，通过 `subprocess.Popen` 的 `stdin.write` 持续向管道内写入指令：

```python
# 初始化阶段（仅执行一次）
adb_proc = subprocess.Popen(
    ["adb", "shell"],
    stdin=subprocess.PIPE, stdout=subprocess.PIPE, stderr=subprocess.PIPE,
    text=True,
)

# 生产者线程内高频调用（无进程创建开销）
def swipe(x1, y1, x2, y2, duration_ms):
    cmd = f"input swipe {x1} {y1} {x2} {y2} {duration_ms}\n"
    adb_proc.stdin.write(cmd)
    adb_proc.stdin.flush()
```

- **Scrcpy 控制 Socket 复用（进阶方案）：** 直接复用 Scrcpy 开源框架留下的物理控制 Socket 端口（Control Socket）进行原始字节注入，彻底绕过 ADB Shell 层，确保滑动轨迹的绝对丝滑和无突刺。

### 6.2 二维高斯热区点击 (Spatial Distribution)

**优化方案：**
当系统计算出点击控件的中心坐标后，严禁使用均匀随机分布。必须引入二维高斯分布（Gaussian Distribution）算法生成最终的随机偏移量。使 95% 的点击点集中在以中心点偏左下方（模拟右手大拇指自然点击习惯）的圆形扩散区域内，越接近卡片边缘，点击概率呈指数级骤降，在风控服务器侧留下完美的类人触控空间热区图。

### 6.3 飞行模式网络矩阵隔离 (反连坐)

**物理有线/无线切换防线：**
严禁设备连接任何固定宽带 Wi-Fi。手机必须插 SIM 卡走蜂窝网络。

**隔离流：**
每当系统轮询完一个店铺、或者在资产池内执行账号切换前，控制端通过 ADB 强制向物理手机下发飞行模式强启指令：

```bash
adb shell svc data disable
adb shell settings put global airplane_mode_on 1
# 强制静置 3 秒，彻底切断与上一个基站的物理附着
adb shell settings put global airplane_mode_on 0
adb shell svc data enable
```

该操作会**迫使移动运营商网络重新为该物理设备分配一个纯净、全新的公网出口 IP**，彻底斩断账号集群间的设备与网络连坐特征。

### 6.4 scrcpy 投屏流与本地 TCP 端口深度隐匿

**工程补丁：源码级特征重构**

- **全面去特征化：** 严禁直接使用开源社区下载的 scrcpy 编译包。必须拉取源码，在编译前将所有涉及 `scrcpy`、`scrcpy-server`、`scrcpy_local` 的字符串、类名、进程名全部批量重命名为毫无特征的系统级包名（例如：`com.android.system.service.impl`）。
- **随机通信隧道：** 每次随动脚本启动拉起投屏时，传输视频流的本地 Local Socket 名称必须使用 UUID 动态随机生成，通信完毕后立即销毁。
- **内核级反检测：** 放弃 Magisk，全面转向 **KernelSU** 或 **APatch**。它们运行在 Linux 内核层，通过修改内核系统调用来隐藏 Root，能完美通过各大电商 App 的深层环境检测（Anti-Taint Audit）。

### 6.5 入店路径全天然伪装 (对抗流量来源审计)

- **搜索混淆：** 脚本指挥手机进入列表页前，先在首页随机点击 1-2 个不相干的热搜词（如"防晒霜"、"男鞋"），假装划动两下。
- **间接进店：** 寻找竞品店铺时，优先在全网搜索该店铺的一款爆款商品名称，在通用的商品搜索结果页中找到该竞品，点击进入其"商品详情页"，再从详情页左下角点击"进店"。将流量来源完美伪装成"自然搜索推荐流量"。

### 6.6 斩断 Scrcpy 本地网络逆向特征（反端口审计 — V5.0 优化）

**风控盲区：**
开源 Scrcpy 默认在手机端拉起一个 Server，并通过 ADB 建立一个网络反向隧道（Reverse Tunnel）将视频流传回 PC。此时手机系统的底层网络状态（`/proc/net/tcp`）中会常态化暴露一个指向本地的监听端口或活动连接。电商平台的顶级安全 SDK（如阿里聚安全）会高频轮询手机的本地套接字，一旦发现异常的反向通信流，直接封号。

**优化规范：正向端口转发 + 本地抽象套接字**

- **严禁默认 `adb reverse` 隧道：** 在控制端拉起 Scrcpy 隐匿流时，必须强制改用 `adb forward`（正向端口转发），或者利用 Android 的 Local Abstract Socket（本地抽象套接字）进行无名管道通信。让手机端在网络层不留下任何非常规的 TCP/IP 通信特征。
- **启动命令改造：**


### 6.7 移动端物理传感器静态熵置换（反静态设备审计 — V5.0 优化）

**风控盲区：**
手机即使平放在桌面上或机柜里，由于地壳微震或环境杂音，其加速度计（Accelerometer）和陀螺仪（Gyroscope）的底层原始数据也必然存在微秒级的正态噪声波动。如果系统在白天连续滑动手机 20 分钟，而手机的传感器数据极其诡异地恒定为绝对零值，风控引擎会一秒判定当前设备为"群控机房平放挂机"，直接施加高阶滑块。

**优化规范：生物级微颤噪声注入**

- **前置条件：** 手机必须已获取 KernelSU 权限。
- **注入策略：** 在预检阶段通过特定模块或系统 Hook，向系统的传感器总线高频注入"生物级微颤噪声"。利用高斯正态分布模拟人手持手机时的呼吸颤动幅值（加速度计 X/Y/Z 轴各自独立注入 `σ ≈ 0.05 m/s²` 的随机抖动）。
- **实现路径：** 通过 KernelSU 模块劫持 `/dev/iio:device*` 或 Sensor HAL 层，在传感器数据上报前叠加正态分布噪声。

### 6.8 Windows 堆管理器物理内存泄漏防御（进程动态生命周期 — V5.0 优化）

**性能盲区：**
Python 的 C 扩展库（如 OpenCV 的 `cv2.imread / imdecode`）在 Windows 环境下运行时，会频繁向系统的 `HeapAlloc` 申请内存。由于 Windows 堆管理器的碎片回收机制比较保守，多线程高频创建和销毁 OpenCV 图像矩阵时，往往会出现"代码层已释放，但系统物理内存（RAM）依然不断缓慢攀升"的假性内存泄漏现象。对于只有 3.6GB 可用物理内存的机器来说，跑半小时就会触发系统剧烈掉帧。

**优化规范：进程动态生命周期模型**

- **废弃单进程常驻模式：** 彻底废弃在单个常驻进程中循环处理图像的做法。
- **店铺级进程隔离：** 当一个店铺采集完毕（触发 `checkpoint.json`）时，主进程主动销毁当前负责 OpenCV 与大模型通信的子进程，并重新 Fork 一个干净的子进程去跑下一个店铺。
- **操作系统级物理回收：** 通过操作系统级的进程退出，强制 Windows 瞬间物理回收该进程占用的所有顽固内存碎片。

```python
import multiprocessing as mp

class StoreProcessManager:
    """店铺级进程生命周期管理 — 每完成一个店铺，销毁旧进程、Fork 新进程。"""

    def __init__(self):
        self._current_process: mp.Process | None = None

    def run_store(self, store_config: dict) -> None:
        # 先销毁上一个店铺的进程，强制 OS 回收所有内存碎片
        if self._current_process is not None:
            self._current_process.terminate()
            self._current_process.join(timeout=10)
        # Fork 全新干净子进程
        self._current_process = mp.Process(
            target=_store_collect_worker,
            args=(store_config,),
        )
        self._current_process.start()
        self._current_process.join()
```

## 7. 异常灾备与人机协作规范 (Human-in-the-Loop)

### 7.1 本地人机解围闸门机制

系统不编写任何复杂的云端打码或自动化破码脚本，全权利用白天手动运行的"人机随动"红利：

1. **挂起：** 当大模型语义解析连续 3 次返回异常，或验证码判定命中时，中枢控制端立即熔断，终止一切手势注入。
2. **唤醒：** 控制端拒绝调用复杂的远程 API，直接驱动电脑本地主板扬声器发出急促的**本地蜂鸣声（Alert Bleep）**，同时控制台高亮红字挂起等待。
3. **复活：** 运营人员直接在手边的物理手机上手动划过滑块（产生最无懈可击的真人生理轨迹），处理完后在电脑键盘上按 `Enter` 键。脚本瞬间清空阻塞队列并复活，继续安全向下抓取。

**兜底策略（无人应答时）：**
若挂起后 **30 分钟** 无 `Enter` 信号，执行硬重置：`am force-stop` 杀进程 → `am start` 重拉 → 重新会话初始化。连续 3 次硬重置失败则彻底停机。

### 7.2 分批断点续跑检查点 (`checkpoint.json`)

允许用户随时因突发事件强行拔线退出：

- 当系统捕捉到控制台的 `Ctrl+C` 中断信号时，生产者立刻停手，消费者将有界队列中剩余的数帧大图清洗入库完毕。
- 随后，系统将当前正在执行的 `store_id`、已采集的商品序号以及最后一条成功写入数据库的 `last_successful_virtual_id` 的序列化快照写入本地目录下的 `checkpoint.json`。
- 当下一次用户插线重启脚本时，预检模块会自动提示用户是否一键恢复历史未完成的断点进度，直接跳过已采集的商品，彻底免除时间浪费。

## 8. 靶点池基础设施配置规范 (`targets.yaml`)

为实现高内聚低耦合，所有的竞品店铺池在本地通过标准 YAML 基础设施进行集中管理，禁止在 Python 核心代码中进行任何硬编码。

```yaml
# 竞品销量纯视觉自动化统计系统 - 靶点池管理配置
global_settings:
  scroll_interval_min: 1.5       # 仿生滑动最小停顿时间(秒)
  scroll_interval_max: 3.5       # 仿生滑动最大停顿时间(秒)
  max_backpressure_queue: 10     # 有界阻塞队列最大帧数
  thermal_limit_celsius: 42.0    # 物理手机核心温度熔断阈值

store_pool:
  # Tier 1: 核心监控店铺（每日必跑，死盯微观订单事件流）
  tier_1:
    - store_name: "赣升江西三支一扶官方旗舰店"
      keywords: ["江西三支一扶公基", "江西三支一扶申论课"]
    - store_name: "皖拓教育官方店"
      keywords: ["综合知识笔试课", "职业能力测试"]
```

系统启动后，Python 调度引擎必须使用 `random.shuffle()` 算法将上述 YAML 列表中载入的店铺靶点顺序完全随机化打散，严禁产生任何固定周期的特征流，使其完美伪装于白天的真实日活流量之中。

## 10. 界面布局改版防护网（UI Shift Kill-Switch）

### 10.1 卡片特征偏离检测

每次采集开始时，通过 OpenCV 对前 5 个商品卡片的宽高比、边界像素分布进行基准校验，建立当前版本 UI 布局的模板基线。

### 10.2 改版数据熔断阀门

| 触发条件 | 阈值 |
|---------|------|
| 连续 N 个商品的视觉特征与系统模板的偏离度 | 超过 15%（连续 20 个商品） |
| 本地大模型连续输出 `{"sales": null}` | 连续 20 次 |
| 动态入口寻址连续未命中 | 连续 10 个商品 |

### 10.3 数据保护

触发系统级**数据强行熔断**：
1. 终止手机滑动和数据库 INSERT 行为。
2. 锁死写入闸门，防止错位脏数据污染历史干净的时序大盘。
3. 向运营团队发送紧急改版告警，等待手动确认新 UI 布局后更新配置。
