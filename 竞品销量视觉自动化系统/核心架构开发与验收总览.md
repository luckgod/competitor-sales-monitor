# 核心架构开发与验收总览

> 项目整体落地划分为五个纵向阶段，每个阶段必须通过"地狱级拦截测试（Chaos Test）"方可进入下一阶段。

## 阶段一：底层通信与链路自愈保障层（环境与骨架）

> 本阶段核心目标是打通物理设备与控制端的无状态连接，彻底解决数据线闪断、进程死锁等物理层致命隐患。

### 1.1 开发执行步骤

**物理环境常驻配置：**
- 配置控制端 ADB 环境变量。
- 调教 Android 测试机：开启"USB调试"、"USB调试(安全设置)"，并在开发者选项中配置"充电时屏幕永不休眠"。

**投屏管道流挂载：**
- 在控制端调用 scrcpy 命令行引擎，指定参数：分辨率 1080P、帧率限流 15fps，将解码后的实时 H.264 画面帧以内存管道（Pipe）形式推送到 Python 进程上下文中。

**断线自愈守护进程（Watchdog）开发：**
- 建立一个独立的常驻健康监控线程。
- 实时轮询 scrcpy 进程状态与 ADB 连接状态。

**运行期断点状态持久化设计：**
- 在内存中建立一个临时的任务状态注册表，用于实时记录当前爬取的"店铺ID"及"最后成功抓取的商品 Index（物理位置锚点）"，为无状态恢复提供数据基础。

### 1.2 单元验收标准

**测试算子（热插拔测试）：**
在系统持续接收视频流的过程中，人为拔掉物理 USB 数据线，等待 5 秒后重新插上。

**判定红线：**
- 控制端主程序**绝对不允许抛出异常退出**。
- 看门狗必须在 **3 秒**内检测到断连，自动在后台执行 `adb kill-server && adb start-server` 重新握手。
- 重连后，自动重新拉起 scrcpy 视频流，并根据内存注册表中的 Index 锚点控制手机自动下滑到断开前的物理位置，实现**无人值守的状态自愈续爬**。

## 阶段二：并发容器与背压控制框架层（骨骼与流水线）

> 本阶段核心目标是解决手机高频截图与本地大模型慢速解析之间的"算力不对称"矛盾，在软件层建立钢筋铁骨。

### 2.1 开发执行步骤

**多线程并发模型搭建：**
显式创建生产者（Producer）与消费者（Consumer）双线程架构。

- **线程 A（生产者）：** 仅负责接收投屏视频流、记录当前帧，并向队列投递。
- **线程 B（消费者）：** 负责从队列中提取图像，进行图像矩阵拆分、去重与 AI 语义清洗。

**有界阻塞队列配置：**
放弃无界队列，强制将核心中转容器设定为 **Bounded Blocking Queue**，最大容量上限死锁为 **10 帧**。

**协同阻塞逻辑（背压控制）实现：**
编写线程同步锁。当队列积压达到 10 帧时，强制让线程 A 进入 **WAITING** 状态，停止一切物理滑动和截图；线程 B 每消费一帧，释放空间后，通过信号量（Notify）即刻唤醒线程 A。

### 2.2 单元验收标准

**测试算子（极限背压与内存压力测试）：**
在线程 B（消费者）中人为注入一行 `sleep(5)` 指令（模拟本地大模型卡死 5 秒场景），同时控制线程 A 疯狂滑动。

**判定红线：**
- 监控控制端 PC 的任务管理器，当队列积压满 10 帧大图（约 60MB 内存占用）后，手机的物理滑动和屏幕截图行为必须**瞬间强制悬停**。
- 系统总内存占用线必须**横向拉平**，严禁无限飙升，严禁触发 OOM（内存溢出）异常。
- 当消费者卡顿结束后，队列开始空出，手机必须在没有任何人工干预的情况下，**自动恢复仿生滑动**。

## 阶段三：视觉解析、模糊去重与两道式清洗网关（核心大脑）

> 本阶段核心目标是完成数据的高精度提取，干掉重复帧、未加载白块，并用最低的算力成本清洗文本。

### 3.1 开发执行步骤

**网格化 OpenCV 卡片剪切：**
编写 OpenCV 算法，对单帧大图执行轮廓检测。按照相对坐标比例，精准切分出每个商品的"主图区（区域A）"、"标题区（区域B）"和"销量区（区域C）"。

**懒加载（Lazy Loading）无效帧过滤：**
在裁剪后，计算区域 A 和区域 C 矩阵的图像信息熵或颜色方差。

**模糊去重引擎（pHash 汉明距离）：**
- 内存开辟最大容量为 **30** 的双端滑动窗口队列。
- 利用 imagehash 算法计算主图的 64 位感知哈希值。
- 新卡片切出后，比对标题相似度。若超过 90%，进一步计算前后两图 pHash 的汉明距离。

**两道式高吞吐清洗网关（Regex-First）：**

| 防线 | 工具 | 处理耗时 | 覆盖比例 |
|------|------|---------|---------|
| 第一道（规则正则） | 标准正则表达式（"月销d+", "d+人付款" 等） | 微秒级 | ~90% |
| 第二道（本地模型兜底） | Ollama Qwen2.5-7B-Instruct (4-bit) | 百毫秒级 | ~10% |

### 3.2 单元验收标准

```text
[ 输入测试数据集 ] -> (OpenCV 卡片拆分) -> (信息熵/方差校验)
  -> (pHash 汉明距离判定) -> (第一道: 正则) -> (第二道: 本地大模型)
```

| 样本 | 输入 | 预期行为 | 入库判定 |
|------|------|---------|---------|
| 样本 1 | 纯白卡片（未加载） | 灰度方差极低，触发无效帧过滤 | 直接销毁，不计入去重队列，LLM 调用 0 次 |
| 样本 2 | 10 帧完全相同商品 | 第 1 帧通过，后 9 帧汉明距离 < 5 被拦截 | 仅入库 1 条，LLM 调用 0 次 |
| 样本 3 | "月销 1.5万+" | 正则完美匹配，代码换算 | 输出 15000，LLM 调用 0 次，耗时 <5ms |
| 样本 4 | "折后券后当前季销2.3W件" | 正则失败，路由至 Ollama | 输出 {"sales": 23000}，num_predict:32 截断 |

## 阶段四：时序分区存储与会话控制层（持久化与安全）

> 本阶段核心目标是解决快照数据的归属问题，并防御海量时序数据写入带来的底层数据库性能危机。

### 4.1 开发执行步骤

**店铺会话初始化（Session Setup）：**
编写前置流控脚本：手机自动点进店铺首页 -> 截取顶部名称框 -> 本地 OCR 识别中文字符串 -> 后端检索并绑定 `store_id` -> 切换至全部商品列表页。

**多账号资产池（Account Pool）切换逻辑：**
编写配置表管理多个小号。设定安全计数器，每完整采集完 N 个店铺，手机脚本强制清空目标 App 缓存，轮询切换至资产池中的下一个小号登录，分散风控压力。

**时序分区数据库（DDL）落地：**
- 严格执行快照历史表建表语句。
- 必须强制附加 `PARTITION BY RANGE (TO_DAYS(record_date))` 字段，实施**按月物理分区**。
- 为 `(virtual_id, record_date)` 设定**联合唯一索引**。

### 4.2 单元验收标准

**测试算子 1（联合索引约束测试）：**
手动向 `rolling_sales_history` 表并发 INSERT 两条 `virtual_id` 和 `record_date` 完全相同的记录。

**判定红线：**
数据库必须拒绝第二条写入，抛出 `Duplicate entry ... for key` 异常，证明联合索引锁死了单日重复跑盘的数据重叠。

**测试算子 2（分区执行计划检查）：**

```sql
EXPLAIN SELECT * FROM rolling_sales_history WHERE record_date = '2026-05-28';
```

**判定红线：**
执行计划中的 `partitions` 字段必须精确指向当前 5 月份物理分区，**严禁**全表全索引扫描（Full Table Scan）。

## 阶段五：全自动调度、高可用运营与全链路联调（全面交付）

> 本阶段核心目标是系统脱离人工后，在各种黑天鹅环境下保持极高的数据可用性。

### 5.1 开发执行步骤

**定时爆破调度器（Cron Job）：**
配置系统常驻守护进程，锁定每日凌晨 **2:00 - 4:00** 自动唤醒设备跑盘，任务结束后彻底杀掉目标 App 进程，整机回归锁屏。

> **Ollama 冷启动饿死隐患补丁：**
> 由于 3.4.1 节实施了正则优先策略，90% 的商品文本在毫秒级被正则处理完，不会触达 Ollama。而 Ollama 默认 5 分钟无请求会自动卸载模型释放内存。
>
> **工程补丁：**
> 1. 凌晨 2:00 定时任务激活瞬间，主控脚本优先向本地 Ollama 发送预热握手请求，强制大模型在跑盘周期内常驻：
> ```bash
> curl -X POST http://localhost:11434/api/generate \
>   -d '{"model": "qwen2.5:7b", "keepalive": -1}'
> ```
> 2. 凌晨 4:00 跑盘结束、杀掉目标 App 后，发送释放指令恢复低功耗：
> ```bash
> curl -X POST http://localhost:11434/api/generate \
>   -d '{"model": "qwen2.5:7b", "keepalive": "0s"}'
> ```

**二次调度机制（Watchdog）：**
编写任务执行状态盘点组件。每日早上 8:30 自动盘点大盘数据。若发现数据整体完整率低于 **90%**（由于半夜断网等原因），任务调度器自动创建补抓任务，会话标签打上 `Replenish_Snapshot` 标记。

**全局改版数据强行熔断锁（UI Shift Kill-Switch）：**
编写校验网关。若系统在运行过程中，发现连续 20 个商品的卡片宽高比例偏离度超过 **15%**，或者本地大模型连续 20 次输出 `{"sales": null}`，判定目标 App 进行了 UI 大改版。

**时序空洞拟合（前端插值消费）：**
编写前端大屏可视化逻辑。在调取 30 天销量折线图时，若发现某天快照因极端原因缺失，前端自动执行线性插值算法进行平滑修复：

```text
Value_target = (Value_date-1 + Value_date+1) / 2
```

并在 ECharts 节点上渲染为**虚线**或特定预警色彩。

### 5.2 全系统终期验收红线（General Gatekeeping）

**地狱测试算子（改版与弹窗综合拦截）：**
在凌晨自动跑盘期间，模拟向手机屏幕上方弹出"系统电量低"或"系统强制升级"的全局遮挡弹窗，或连续传入 20 张全黑、错位的商品卡片。

**判定红线（全自动化与稳定性终极指标）：**

1. **绝对不允许**主进程崩溃抛出异常而停止工作。
2. 改版数据强行熔断锁必须在第 **20 帧**异常发生后**毫秒级**触发。
3. 脚本必须立刻强制终止手机滑动，**死锁数据库写入闸门**（不写入任何一条错位脏数据）。
4. 系统必须保持**挂起**状态，通过钉钉/企业微信机器人向管理员发送最高级别告警通知：

> "UI 布局可能改版/屏幕遭受系统弹窗严重污染，系统已启动紧急自我保护熔断锁"
